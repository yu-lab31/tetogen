# tetogen
Turn go.lang structs into .yaml config files.

Tetogen is a lightweight configuration template generator for Go projects, completely driven by AST (Abstract Syntax Tree) static analysis.

By scanning specific magic comments (// +tetogen:) across your codebase, it automatically scaffolds multi-environment .yaml files, alongside a matching .schema.json. This gives your IDE/LSP (e.g., VS Code, Neovim) instant type validation, autocompletion, and default value hints directly inside your YAML files.

## Features

- Zero Runtime Dependencies: Powered entirely by go/parser and go/ast. No runtime overhead, no reflection, and zero bloat to your production binary.
- Multi-Profile Automation: Generate dev, test, prod, or any customized environment configurations with a single command.
- Robust Type Mapping: Full support for primitive types, nested structs, arrays/slices, and dynamic maps.
- LSP Integration out of the Box: Generates clean YAML and injects the yaml-language-server directive to instantly connect your YAML files with their schema.
- Root-Anchored Execution: Automatically resolves up the directory tree to find your go.mod. Run it from any subdirectory, and your configuration files will always land exactly where they belong.

## Quick Start

### Add Magic Comments

Add the `// +tetogen:` magic comment directly above your target configuration struct:

```go
package config

// +tetogen:profiles=dev,test,prod out=configs/
type AppConfig struct {
	Server struct {
		Port  int      `yaml:"port" env-default:"8080"`
		Hosts []string `yaml:"hosts" env-default:"127.0.0.1,192.168.1.1"`
	} `yaml:"server"`
	
	Metadata map[string]string `yaml:"metadata"`
}
```

### Install and Run the Generator

1. Install by:

```shell
go install github.com/yu-lab31/tetogen@latest
```

2. Run the script from anywhere within your repository (you can also bind this to your Makefile):

```shell
tetogen
```

### Find the Output

You will find:

```text
configs/
├── appconfig.schema.json      # Single source of truth validation schema
├── appconfig_dev.yaml         # Dev template with injected LSP headers
├── appconfig_prod.yaml        # Prod template
└── appconfig_test.yaml        # Test template
```

under `$HOME` of your project. You can change the path where config files are generated by modifying `out=...` command in the magic comment.

`tetogen` generates `appconfig.yaml` and `schema.json` only when `profiles=...` is not specified.

## Design Philosophy

### Ignoring Cross-Package Struct Nesting

Nesting massive external configurations (like redis.Options or gorm.Config) directly inside a static config manifest is highly discouraged:

- Tag Pollution: Third-party runtime structs should not be polluted with environment-specific configuration engine tags (e.g., env-default).
- Clean Architecture / Separated Concerns: Configuration declarations should act as a lightweight, human-readable manifest.

but most importantly, checking these cross-package structs with static analysis is very hard, so I simply ignore them haha.
